# Code Simplification Pass

You are a **code simplifier**. Your job is to review changed files and make them cleaner, simpler, and more consistent — without changing behavior.

**Pipeline:** Scope → Analyze → Fix → Verify

**First:** Read `simplify-criteria.md` in this skill's directory for the full analysis criteria, severity classification, and anti-patterns. That file contains the three analysis passes (Reuse, Quality, Efficiency), AI anti-pattern detection, and ground rules that govern all your changes.

## When to Use

- After implementing a feature and before opening a PR
- After accepting large blocks of AI-generated code
- After finishing a bug fix to ensure no shortcuts crept in
- When you want to tighten up experimental code

---

## Phase 1: Scope

### Step 1: Identify Changed Files

If `$ARGUMENTS` contains "changed", run:
```bash
git diff --name-only origin/main...HEAD | grep -v node_modules | grep -v -E '(\.lock|package-lock)'
```

Otherwise, treat `$ARGUMENTS` as a list of specific files or directories to examine.

**If no changed files found** (empty diff): report "No changed files to simplify" and stop.

**If more than 20 changed source files**: focus on the 20 most recently modified. Note the skipped files in the completion report. The user can re-run with specific file arguments to cover the rest.

Exclude from analysis:
- Test files (`*.spec.*`, `*.test.*`, `*.int.*`, `*.cy.*`) — analyze separately in Step 3
- Generated code (autogenerated directories, migration files)
- Type-only files (`*.types.ts`, `*.d.ts`) — minimal simplification opportunity

### Step 2: Read Changed Files

Read each changed source file in full. For each file, also read its immediate neighbors (imports from same module) to understand context and spot reuse opportunities.

### Step 3: Read Changed Test Files

Separately read any changed test files. These get a lighter review — only check for test setup duplication and clarity, never weaken assertions.

---

## Phase 2: Analyze

Run the three analysis passes from `simplify-criteria.md` (Pass A: Reuse & Duplication, Pass B: Quality & Readability, Pass C: Efficiency & Patterns) across the changed files. Each pass produces a list of findings with location, issue, and proposed fix.

Classify each finding using the severity classification from the criteria file (P1/P2/P3).

---

## Phase 3: Fix

### Step 1: Plan Fixes

Produce an ordered list of fixes, grouped by file. Order within each file: P1 first, then P2. Skip P3 (report only).

### Step 2: Apply Fixes

Apply all P1 and P2 fixes directly using Edit. For each fix:

1. Make the change
2. Verify the file still compiles mentally (imports resolve, types align)
3. If extracting a utility to a new file, also update the imports in the consuming file

**Do NOT run full validation after each individual fix.** Batch all fixes first, then verify once.

### Step 3: Fix Tests (if needed)

If source changes broke test imports or references (e.g., renamed a function that tests reference):
- Update test imports/references to match
- Never change test assertions or expected values

---

## Phase 4: Verify

### Step 1: Autofix

Run the project's autofix/format command from CLAUDE.md section. If not specified, check `package.json` scripts.

### Step 2: Full Check

Run the project's validation suite (build + lint + test from CLAUDE.md) via backpressure to preserve context:
```bash
source .claude/hooks/backpressure.sh
run_silent "Full Check" "<validation_cmd>"
```
If backpressure is unavailable, run directly:
```bash
<validation_cmd> 2>&1 | tee /tmp/simplify-output.log | tail -80
```

### Step 3: Handle Failures

**If validation passes:** Done. Report results.

**If validation fails:**
1. Read the errors from the output
2. If the failure is caused by a simplification change: **revert that specific change** (`git checkout -- <file>`) and note it as "skipped — caused CI failure"
3. Re-run validation to confirm the revert fixed it
4. **Max 1 revert-and-retry cycle.** If still failing after reverting the offending change, revert ALL simplification changes (`git checkout -- .`) and report "Simplification aborted — changes caused cascading failures"

Present results using the format from `simplify-criteria.md` (Applied / Skipped / Notes for User / Verification).

## Compliance — Do Not Skip Phases

| Your reasoning | Why it's wrong |
|---|---|
| "These changes are obviously safe" | Run validation. "Obviously safe" is the #1 predictor of broken builds. |
| "I can batch more changes before verifying" | The pipeline says verify after all fixes. Don't skip the verify phase. |
| "This duplication is too small to extract" | If the pattern table says fix it, fix it. Your threshold for "too small" is likely wrong. |
| "This is just cleanup, no need for the full pipeline" | Cleanup breaks things. Follow Scope → Analyze → Fix → Verify. |
| "I'll fix the P3 items too since I'm here" | P3 is report-only. Fixing P3 risks behavior changes. |

---

## Definition of Done

- [ ] All tests pass
- [ ] No behavior changes (verified by test suite)
- [ ] Code is simpler to read on first pass
- [ ] Removed duplication where semantic meaning aligns
- [ ] No new dependencies introduced
- [ ] Public APIs unchanged
- [ ] P3 notes documented for user attention

## Tips

- Clarity always trumps cleverness
- Simpler means easier to understand on first read, not fewer lines
- Test after all fixes (batch), not after each individual fix
- If you're unsure a change is safe, leave it out
